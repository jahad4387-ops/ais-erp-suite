# 个人优先、预留多人协作的开发方式

## 1. 当前阶段定位

当前项目先按个人开发推进，但从第一天就保留多人、多 Agent 并行开发的结构。目标是：一个人开发时足够轻，不被流程拖慢；未来加入开发者或多个 Agent 时，模块边界、接口和验收规则已经清楚，不需要推倒重来。

当前推荐模式：

```text
个人主导开发
-> 每个阶段一个清晰模块
-> 每个模块有接口和验收
-> 重要变更走分支和提交
-> 未来自然扩展为多人协作
```

## 2. 开发原则

1. `main` 始终保持可运行。
2. 每次只做一个明确主题。
3. 模块边界先写清楚，再写代码。
4. 核心领域规则放在共享包，不在各模块重复实现。
5. API 契约先于前后端联调。
6. Agent 可以参与开发，但必须明确文件范围和任务边界。

## 3. 模块边界

| 模块 | 当前个人开发阶段 | 未来多人分工 |
| --- | --- | --- |
| 平台与系统管理 | 优先开发 | 平台组 |
| 总账 | 优先开发 | 总账组 |
| 采购与付款 | 第二批开发 | 采购应付组 |
| 销售与收款 | 第二批开发 | 销售应收组 |
| 存货核算 | 第三批开发 | 存货组 |
| 薪资 | 第四批开发 | 薪资组 |
| 固定资产 | 第四批开发 | 固资组 |
| 报表 | 第五批开发 | 报表组 |
| AI / Agent | 贯穿开发，先做基础设施 | Agent/AI 组 |

## 4. 个人开发时的推荐节奏

个人开发不要同时打开太多战线。推荐按这个顺序：

1. 平台基础：账套、用户、权限、会计期间。
2. 总账 MVP：科目、凭证、审核、记账、账簿。
3. Agent 基础设施：Agent Action、dry-run、审计日志。
4. 采购付款和销售收款。
5. 存货和成本。
6. 薪资和固定资产。
7. 报表和经营分析。
8. 深度 AI 自动化。

每个阶段完成后再进入下一个阶段，避免系统半成品太多。

## 5. 分支策略

当前个人开发可以保持简单：

```text
main
feature/platform-mvp
feature/general-ledger-mvp
feature/agent-gateway
feature/procure-to-pay
feature/order-to-cash
```

规则：

- 小文档、小修复可以直接提交到 `main`。
- 新模块、数据库结构、核心规则改动使用 `feature/*` 分支。
- 每个分支只做一个主题。
- 合并前必须跑测试。
- 合并后 `main` 应该能启动或至少能通过领域层测试。

未来多人开发时：

- 每个人或每个 Agent 使用独立分支。
- 通过 Pull Request 合并。
- 每个 PR 写清楚影响模块、测试结果、风险点。

## 6. 模块契约

每个模块开始开发前，先补一页模块契约。格式如下：

```text
模块名称：
负责范围：
不负责范围：
依赖模块：
对外提供 API：
会调用的 API：
会生成的凭证：
核心数据表：
权限点：
审计点：
验收用例：
```

示例：总账模块。

```text
模块名称：总账
负责范围：科目、期初、凭证、审核、记账、账簿、期末处理
不负责范围：采购订单、销售订单、库存出入库、工资计算、固定资产折旧
依赖模块：平台、基础档案、审批、审计
对外提供 API：创建凭证草稿、提交凭证、审核凭证、记账、查询账簿
会调用的 API：权限校验、期间校验、审计日志
会生成的凭证：手工凭证、结转凭证、冲销凭证
核心数据表：accounts、vouchers、voucher_lines、ledger_entries
权限点：凭证录入、凭证审核、凭证记账、反记账、结账
审计点：凭证创建、修改、审核、记账、冲销、作废
验收用例：借贷平衡、已记账不可删除、试算平衡、期间关闭后不可记账
```

## 7. 接口优先

多人协作最容易卡在“你还没做完，所以我没法做”。解决办法是先定义接口。

推荐流程：

```text
先写 OpenAPI / 接口草案
-> 前端和 Agent 按 mock 数据开发
-> 后端按接口实现
-> 测试按接口验收
```

接口变更规则：

- 新接口可以逐步增加。
- 删除或改字段必须谨慎。
- 影响其他模块的接口变更要写在 PR 描述里。
- Agent 相关接口必须保留 `dryRun`、`idempotencyKey`、`evidenceRefs`。

## 8. Agent 参与开发的规则

未来如果用多个 Codex、Claude Code 或其他 Agent 并行开发，要明确分工。

好的任务写法：

```text
你负责 packages/domain/src/ledger.mjs 和 packages/domain/tests/ledger.test.mjs。
实现凭证状态机和测试，不要修改采购、销售、报表文件。
```

不好的任务写法：

```text
帮我把财务系统做完。
```

Agent 分工建议：

| Agent | 适合任务 |
| --- | --- |
| Explorer Agent | 阅读代码、总结模块边界、找风险 |
| Worker Agent | 实现某个小模块或测试 |
| Reviewer Agent | 检查 PR、找 bug、查遗漏 |
| Documentation Agent | 整理 PRD、接口文档、验收清单 |

Agent 也必须遵守：

- 不改无关文件。
- 不回滚别人改动。
- 不跳过测试。
- 不在没有审批的情况下改核心规则。
- 改动必须说明文件路径和验证结果。

## 9. PR 规则

当前个人开发可以不强制每次开 PR，但未来多人开发建议使用。

PR 模板建议：

```text
## 变更内容

## 影响模块

## 测试结果

## 风险点

## 是否影响 API

## 是否影响 Agent 操作

## 是否需要数据迁移
```

PR 合并前检查：

- 测试通过。
- 文档同步更新。
- API 变更已记录。
- 高风险财务规则经过复核。
- 没有提交构建产物、日志、临时文件。

## 10. 验收方式

每个模块至少要有三层验收：

1. 单元测试：核心规则正确。
2. 集成测试：模块之间能串起来。
3. 业务场景测试：真实工作流能跑完。

示例：总账 MVP 验收。

```text
创建账套
创建会计期间
创建科目
录入借贷平衡凭证
提交审核
审核通过
正式记账
查询明细账
查询试算平衡表
关闭期间
验证关闭期间不能继续记账
```

## 11. 当前最适合的执行方式

短期内推荐你保持个人主导：

```text
1 个主开发者
+ 1 个 Codex 辅助实现
+ 1 个测试/审查习惯
+ 清晰模块文档
```

也就是说，现在不需要马上搭复杂团队流程。先把平台和总账做稳，再逐步引入更多 Agent 或开发者。只要模块契约、接口和测试一直保留，未来多人开发就会自然接上。

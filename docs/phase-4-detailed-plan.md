# Phase 4：薪资与固定资产详细开发计划

这一阶段承接 Phase 1 的总账、Phase 2 的应收应付、Phase 3 的存货成本，目标是把企业常见的 **人工成本** 和 **长期资产折旧** 纳入自动核算闭环。

# Phase 4 总目标

Phase 4 要建立两个核心闭环：

```text
薪资闭环：
员工档案 → 薪资类别 → 工资项目 → 考勤/绩效/补贴扣款导入 → 工资计算 → 工资审核 → 工资发放 → 工资费用分摊 → 自动生成总账凭证 → 工资台账 / 部门人工成本 / 总账对账

固定资产闭环：
资产类别 → 固定资产卡片 → 新增资产 → 部门/使用状态管理 → 折旧计提 → 资产转移 / 原值变动 / 处置 → 自动生成总账凭证 → 固定资产明细账 / 折旧表 / 总账对账
```

# 4. 薪资业务流程与防坑设计

## 4.1 薪资初始化与个税累计壁垒

> [!CAUTION]
> **同月多批次发放的“个税累计壁垒”**
> 系统必须支持多发薪批次（如本月基本工资批次、奖金批次）。在个税引擎规则设计时明确：同一个自然月（税款所属期）内的多次工资发放，个税公式**必须能够自动汇总该月该员工所有已发批次的应税收入**，实行累计计税，严禁孤立计税导致免税额度被重复扣除的逃税风险。

工资项目及公式引擎：
- 必须可动态配置。`应发工资 = 基本工资 + 绩效 + 导入计时计件工资`。

## 4.2 工资计算流程与尾差微调

> [!NOTE]
> **社保/个税的手工微调干预**
> 由于官方社保账单经常出现几分钱或几块钱的四舍五入尾差，**系统必须允许在工资批次进入审核和锁定之前，财务/HR 能够手工覆写特定员工的社保、公积金和个税系统计算金额**。不允许修改将导致银行流水与系统账务永久无法对齐。

流程：
```text
导入考勤/计件数据 → 创建计算批次 → 执行公式 → 人工微调（应对尾差） → 财务审核 → 锁定批次
```

## 4.3 工资发放、分摊与期末

- **发放**：审核后生成银行代发文件，确认发放后生成：`借：应付职工薪酬 贷：银行存款`。
- **分摊**：按员工所属部门生成：`借：管理/销售/研发费用 贷：应付职工薪酬`。
- **期末**：月底必须通过月末检查，确保本期工资全部审核分摊并生成凭证后，方可关闭工资期间。

# 5. 固定资产业务流程与防坑设计

## 5.1 固资初始化与新增资产

> [!CAUTION]
> **新增/减少资产的“折旧时间线”**
> 必须硬编码该时间线红线：**当月新增的固定资产，当月绝对不提折旧，从下月起开始计提折旧；当月减少（处置/报废）的固定资产，当月照样计提最后一次折旧，从下月起停止计提**。这是通用会计准则，不可违背。

## 5.2 资产转移

> [!IMPORTANT]
> **资产月中转移的“降维处理”**
> 如果资产在某月 15 日发生跨部门转移，为避免研发陷入极其复杂的按天加权平摊费用算法中，Phase 4 MVP 明确规定：**无论资产当月转移多少次，当月的折旧费用全额归属于该月最后一天资产所在的使用部门**。

## 5.3 折旧计提与“强力刹车”

折旧计提过程必须支持 `dry-run` 和人工审核。

> [!CAUTION]
> **提足折旧后的强力刹车机制**
> 绝对不允许资产出现负数净值！在底层折旧引擎中必须增加硬性兜底拦截校验：在每次计提时，如果 `(累计折旧额 + 本次拟提折旧额) >= (原值 - 预计净残值 - 减值准备)`，**系统只能提取剩余的差额作为本次折旧额，并强制将该资产标记为停止后续折旧计算**，避免系统变更为“幽灵资产”。

## 5.4 资产处置与盘点

处置流程支持：出售、报废、盘亏。
凭证链路：原资产转入“固定资产清理”科目，收到处置款转入清理科目，最后将清理科目余额结转至“资产处置损益”科目。

# 7. 总账凭证集成设计与 API

通过配置 `payroll_account_mapping_rules` 和 `fixed_asset_account_mapping_rules`，系统需自动将业务单据生成对应的记账凭证草稿（如工资计提、折旧费用、资产处置损益）。

API 设计依然遵循严格的 `submit -> approve -> generate-voucher` 状态机流转模式，确保业财数据步调一致。

# 9. 权限与 Agent 边界

**Agent / AI 设计**：
Agent 工具：`analyze_payroll_exceptions`、`suggest_payroll_allocation`、`run_depreciation_dry_run`。
**红线**：AI/Agent 默认只能生成工资异常检查报告、折旧测算 `dry-run` 结果和草稿。正式记账、发放工资等涉及资金或正式总账的高危动作，必须经过人工审批。

---

# 10. 开发排期：12 周详细版

- 第 1-3 周：数据底座、薪资基础设置、考勤/绩效导入与工资计算（支持手工调整和个税累计）。
- 第 4-5 周：工资审核、锁定、代发文件、费用分摊、生成总账凭证。
- 第 6-7 周：资产类别、期初卡片录入、资产新增与变动（明确部门归属日）。
- 第 8-9 周：折旧计提引擎（加入刹车机制与时间线控制）、工作量/平均年限折旧。
- 第 10-11 周：资产处置、盘点、固资明细账与折旧凭证集成。
- 第 12 周：月末检查联调、对账测试（工资台账/固资台账核对总账）、验收报告。

---

# 11. 一句话总结

Phase 4 的核心是把企业的 **人工成本** 和 **长期资产折旧** 纳入可计算、可审核、可记账、可对账的财务闭环，并提前封堵所有违背会计准则的算法漏洞。

```text
薪资：多次发放并个税 + 手工尾差允许 → 分摊凭证正确 → 工资台账与总账一致
固资：下月折旧时间线 + 净值刹车兜底 → 折旧凭证正确 → 资产净值与总账一致
```

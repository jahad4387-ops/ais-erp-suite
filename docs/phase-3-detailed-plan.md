# Phase 3：存货、生产与成本核心闭环详细开发计划

Phase 3 的定位从纯粹的“商贸进销存”升级为 **“库存数量管理 + 生产制造转换 + 成本叠加归集 + 存货账表 + 与总账一致性”**。

最终闭环是：

```text
存货档案 (含 BOM 物料清单)
→ 仓库 / 货位 / 批次
→ 采购入库 / 销售出库 / 其他出入库 / 调拨 / 盘点
→ 生产工单 (Work Order)
→ 生产领料出库 (耗用原材料)
→ 归集直接人工与制造费用 (来自 Phase 4 薪资与折旧)
→ 月末成本分摊引擎 (Cost Allocation)
→ 产成品完工入库 (叠加总成本)
→ 成本凭证生成
→ 存货余额与总账一致
```

# 3. 模块契约与数据模型

核心表结构扩展：
```text
存货与生产基础：
inventory_items (增加 is_manufactured 标记, cost_method 等)

work_orders (工单：开工时间、预计产出量)
material_requisitions (生产领料单：消耗原材料库存，计入直接材料成本)

成本归集：
cost_allocations (月末成本分摊表：把 Phase 4 的人工和折旧费用按算法砸进完工入库单里)
```

# 4. 核心业务流程与成本防坑设计

## 4.1 生产制造成本归集流程

> [!CAUTION]
> **月末成本归集的顺序死锁**
> 加入生产制造后，月末结账必须严格遵循时间线红线：
> 1. 必须先锁死当月全部**领料单**，计算出原材料出库成本（直接材料）。
> 2. 必须等待 Phase 4 算完当月**生产工人工资**和**车间设备折旧**（归集人工与制造费用）。
> 3. 最后运行 Phase 3 的**“工单成本分配引擎”**，将这三部分成本按工时/产量比例砸进产成品的“完工入库成本”中。
> **顺序一旦颠倒，成本永远算不平。**

## 4.2 零数量金额余值结转

> [!CAUTION]
> **数量为0但金额不为0的“余值死锁”**
> 采用移动平均等方法时，只要系统判定某存货数量变为 `0`，必须自动生成一笔成本调整记录，将残余库存金额（如暂估尾差）全部结转入当期主营业务成本或损益科目，确保“数量为 0 时，金额一定为 0”。

## 4.3 批次与平均法的互斥

> [!IMPORTANT]
> **批次管理与成本计价方法的互斥**
> 凡是开启了“批次管理”的存货，成本计价方法应强制限定为“个别计价法”或“FIFO”。严禁对批次存货跑全局“移动平均法”。

# 5. 开发排期：12 周详细版

- 第 1-3 周：存货档案、批次序列号、BOM结构设计、库存期初。
- 第 4-6 周：常规出入库、调拨、盘点、移动平均/FIFO核心算法。
- 第 7-9 周：生产工单、领料出库、完工入库逻辑打通。
- 第 10-11 周：打通 Phase 4 费用（前期使用 Mock 数据），实现月末生产成本分摊引擎与“零数量余值结转”。
- 第 12 周：生成总账成本凭证、库存明细账对账、端到端联调验收。

# 6. 一句话总结

升级后的 Phase 3 完美折叠了“商贸进销存”与“工业制造成本”：

```text
准确的库存数量
+ 严密的工单领料完工转换
+ 融合薪资与折旧的制造成本分摊
+ 防坑到底的出库算法兜底
= 毫厘不差的业财一体化存货总账
```

# Phase 3：存货与成本详细开发计划

Phase 3 要把 Phase 2 里已经存在的采购入库、销售出库，升级成真正的 **库存与成本核算系统**。核心定位是：**库存数量管理 + 出入库单据 + 成本计算 + 库存账表 + 与总账一致性**。

最终闭环是：

```text
存货档案
→ 仓库 / 货位 / 批次 / 计量单位
→ 库存期初
→ 采购入库 / 销售出库 / 其他出入库 / 调拨 / 盘点
→ 库存数量账
→ 成本计算
→ 出库成本确认
→ 成本凭证生成
→ 收发存、库存明细账、库龄、周转分析
→ 存货余额与总账一致
```

Phase 3 的核心不是“多做几个库存页面”，而是建立：
- 数量流：单据 → 库存余额
- 成本流：入库成本 → 成本层 → 出库成本 → 成本凭证
- 财务流：库存业务 → 总账凭证 → 存货总账一致

# 4. 核心业务流程

## 4.1 采购入库到成本确认

```text
采购订单 → 采购入库单 → 库存数量增加 → 形成入库成本来源 → 采购发票到票 → 入库成本确认或调整 → 生成存货入账凭证 / 暂估回冲凭证
```

> [!CAUTION]
> **采购暂估价差（PPV）的分摊边界**
> 当发票确认的实际成本与暂估入库成本不一致产生价差时，如果存货使用移动平均等实时算法，价差**只能按比例分摊给当前尚未出库的剩余数量**。对于已经出库对应的价差部分，直接计入主营业务成本或材料成本差异科目，绝不允许将全部价差强行塞给极少数的剩余库存，导致成本单价畸高。

## 4.2 销售出库到成本结转

```text
销售订单 → 发货单 / 销售出库单 → 库存数量减少 → 根据成本方法计算出库成本 → 生成销售成本结转凭证
```

## 4.3 调拨流程与盘点流程

(按原计划：支持仓间调拨不改总金额，盘盈盘亏走待处理财产损溢或管理费用)

# 5. 数据模型设计

## 5.1 存货档案

> [!IMPORTANT]
> **批次管理与成本计价方法的互斥控制**
> 在系统底层配置时，凡是开启了“批次管理”或“序列号管理”的存货，其成本计价方法应被强制限定为“个别计价法”或“FIFO”。严禁对开启批次管理的存货跑无视批次的全局“移动平均法”，以免业务账与财务账彻底脱节。

```text
inventory_items
- id
- cost_method: moving_average / fifo / specific / monthly_weighted_average
- enable_batch
- enable_serial
- enable_shelf_life
...
```

## 5.2 出入库单据与余额表

- `stock_movements`, `stock_movement_lines`
- `stock_balances`
- `cost_layers` (用于 FIFO 或个别计价)
- `cost_calculation_runs` (成本运行记录)

# 6. 成本方法与防坑底线设计

## 6.1 移动平均

公式：`(当前库存金额 + 本次入库金额) / (当前库存数量 + 本次入库数量)`。

> [!CAUTION]
> **负库存与平均法的致命冲突**
> 在 Phase 3 MVP 中，针对采用“移动平均法”和“月末一次加权法”的存货，系统底层**严格禁止开启允许负库存出库**的账套参数。负库存会导致平均单价计算的分母为 0 或计算结果扭曲（负数金额除以负数数量等情况）。

## 6.2 FIFO 先进先出与个别计价

按入库时间或指定序列从早到晚消耗 `cost_layer`。

## 6.3 月末一次加权平均

月末加权平均单价 = `(期初库存金额 + 本期入库金额) / (期初库存数量 + 本期入库数量)`。

> [!NOTE]
> **月末加权出库凭证生成时机**
> 采用该方法的存货，其日常销售出库单**严禁实时生成**主营业务成本凭证。必须等到月末执行 `cost_calculation_runs` 后，系统回填实际出库单价，然后再批量生成该期间的出库总账凭证。

## 6.4 防坑：数量为0但金额不为0的“余值死锁”

> [!CAUTION]
> **零数量余值自动结转机制**
> 在所有的成本核算方法（尤其是移动平均）中，只要系统判定某仓库某存货的数量变为 `0`，系统必须立即自动生成一笔成本调整记录，将残余的库存金额（如 0.05 元或 -200 元暂估尾差）**全部结转入当期主营业务成本或损益科目**，确保“数量为 0 时，金额一定为 0”，防止余值污染下一次入库单价。

# 9. 权限、Agent 与验收要求

## 9.1 Agent 边界

Phase 3 的 Agent 只能做：库存分析、补货建议、异常预警、计算 `dry-run` 和提交审批。绝对不能绕过系统底层去强行修改成本或强制结账。

## 9.2 测试与对账要求

```text
库存余额数量 = 期初数量 + 入库数量 - 出库数量
库存余额金额 = 期初金额 + 入库金额 - 出库成本 ± 调整金额
库存明细账金额 = 总账存货科目辅助余额
```

# 10. 开发排期：12 周详细版

- 第 1-3 周：数据底座、存货档案、批次序列号、库存期初。
- 第 4-7 周：出入库单据、调拨、盘点。
- 第 8-10 周：移动平均、FIFO、加权平均、成本调整。
- 第 11-12 周：生成总账凭证、对账、报表、端到端联调验收。

# 11. 一句话总结

Phase 3 的核心是把 Phase 2 的采购入库、销售出库真正变成 **可核算、可复核、可对账的库存成本系统**：

```text
库存数量正确
+ 出库成本可复核（防坑公式兜底）
+ 库存明细账正确
+ 成本凭证可追溯
+ 存货余额与总账一致
```

# AIS ERP Suite UI 与页面设计规划

## 1. 设计目标

AIS ERP Suite 的界面服务于财务、业务、管理和受控 Agent 协作。页面设计不追求展示型视觉效果，而追求清晰、稳定、低干扰和高频操作效率。

核心目标：

1. 让财务人员可以快速录入、审核、查询和对账。
2. 让管理人员可以稳定查看经营数据、异常和待办。
3. 让系统管理员可以清楚配置账套、权限、部署和审计。
4. 让 Agent 相关动作有明确的 dry-run、证据、风险、审批和回放界面。

## 2. 视觉原则

本系统采用简洁商务风格。

- 信息密度高，但层级清楚。
- 使用中性色为主，少量状态色表达风险和流程。
- 不做营销式大标题、装饰插画、渐变背景或夸张卡片。
- 表格、筛选、表单和工作台优先，适合长时间办公。
- 页面宽度优先服务横向表格和凭证录入。
- 重要操作必须可见，但危险操作不抢视觉焦点。

推荐视觉基调：

| 项目 | 建议 |
| --- | --- |
| 主色 | 深青蓝或企业蓝，用于主按钮、选中状态和导航高亮 |
| 辅助色 | 灰、浅蓝灰、白色背景，用于工作区和表格 |
| 状态色 | 绿色通过、黄色待处理、红色异常、蓝色信息 |
| 字体 | 系统字体优先，中文使用 Microsoft YaHei / PingFang SC |
| 圆角 | 控件和卡片控制在 4px-8px |
| 间距 | 以 8px 为基础栅格，表格和表单采用紧凑尺寸 |

## 3. 全局布局

### 3.1 桌面端主布局

```text
---------------------------------------------------------------+
| 顶部栏：产品名 / 当前账套 / 当前期间 / 命令搜索 / 用户菜单          |
+--------------+------------------------------------------------+
| 左侧模块导航   | 页面标题 + 面包屑 + 当前页面动作                   |
|              +------------------------------------------------+
| 平台          | 筛选区 / 状态概览 / 主数据区                       |
| 总账          |                                                |
| 采购付款       | 表格、凭证、单据、账簿、工作台、抽屉详情              |
| 销售收款       |                                                |
| 存货成本       |                                                |
| 薪资固资       |                                                |
| 报表          |                                                |
| Agent         |                                                |
+--------------+------------------------------------------------+
```

主布局要求：

- 顶部栏固定显示当前账套、组织和会计期间。
- 左侧导航按业务模块分组，支持折叠。
- 页面主区域不使用大面积欢迎页，默认进入可操作工作台或列表。
- 所有列表页统一采用“筛选区 + 表格 + 批量操作 + 分页”结构。
- 所有详情页统一采用“状态条 + 主信息 + 明细表 + 审计/附件/来源”结构。

### 3.2 响应式边界

ERP 高频操作默认以桌面端为主。移动端只支持审批、查看待办、查看报表摘要和异常确认，不承载凭证录入、报表设计器、批量对账等复杂任务。

## 5. 通用页面模式

### 5.1 列表页

适用：凭证列表、采购订单、销售订单、工资批次、资产卡片、报表运行、Agent 动作。

结构：

```text
页面标题 / 主动作
筛选条：期间、状态、编号、往来单位、制单人、金额范围、更多筛选
状态标签：全部 / 草稿 / 待审核 / 已审核 / 已记账 / 异常
表格：编号、日期、对象、摘要、金额、状态、制单人、更新时间、操作
底部：分页、批量操作、导出
```

要求：

- 表格列可隐藏、排序和固定。
- 高风险状态用状态标签和文字同时表达，不只依赖颜色。
- 批量操作必须先进入预览确认，不直接执行。
- 空状态直接给出下一步动作，如“新建凭证”“导入银行流水”。

### 5.2 编辑页

适用：凭证录入、单据录入、工资明细调整、资产卡片、报表模板。

结构：

```text
顶部：保存、提交、预览凭证、附件、更多
状态条：草稿 / 已提交 / 已审核 / 已记账 / 已作废
主表单：基础信息、业务对象、日期、期间、部门、经办人
明细表：行项目、科目、辅助核算、数量、金额、税额
右侧抽屉：附件、审批记录、AI 建议、审计日志
```

要求：

- 保存和提交分开。
- 所有系统生成项显示来源和规则。
- 校验错误定位到具体字段或明细行。
- 离开未保存页面时提醒。

### 5.3 工作台

适用：审核、记账、银行对账、付款审批、折旧计提、成本计算、结账检查。

结构：

```text
左侧：任务队列和筛选
中间：待处理对象列表
右侧：预览、差异、风险、操作确认
底部：批量动作结果和审计提示
```

要求：

- 工作台优先支持键盘和批量处理。
- 执行动作前展示影响范围、金额合计、凭证数量、异常数量。
- 所有提交类动作显示“将产生什么结果”。

### 5.4 详情页

适用：凭证详情、单据详情、报表详情、Agent Action 详情。

结构：

```text
状态条 + 关键金额 + 主体信息
明细表 / 分录 / 单元格结果
来源链路：来源单据 -> 凭证 -> 账簿 -> 报表
附件与证据
审批与审计时间线
```

要求：

- 已记账、已锁定、已归档对象默认只读。
- 反向操作使用独立入口，如冲销、反审核申请、重算申请。
- 详情页必须提供稳定 URL。

## 8. 状态与错误设计

### 8.1 页面状态

所有页面至少支持：

- 加载中。
- 无数据。
- 有数据。
- 权限不足。
- 当前期间关闭。
- 账套未启用。
- 接口失败。
- 正在被他人编辑。

### 8.2 财务错误表达

错误提示必须同时说明：

1. 哪个对象出错。
2. 为什么出错。
3. 用户下一步可以做什么。
4. 是否影响正式数据。

示例：

```text
凭证不能提交：借方合计 10,000.00，贷方合计 9,800.00，差额 200.00。
请检查第 3 行和第 4 行分录金额。当前凭证仍为草稿，未影响账簿。
```

## 9. Agent-friendly 要求

所有页面必须遵循 `docs/agent-native-design.md` 的 UI 要求，并补充：

- 页面路由稳定，不使用只依赖弹窗状态的关键流程。
- 主要按钮必须有稳定 `data-testid`。
- 表格行操作必须可通过对象 ID 定位。
- 批量操作必须有预览页。
- 审批和执行动作必须显示风险等级。
- Agent dry-run 结果与人工执行结果使用同一套差异展示组件。
- Agent 对话窗口必须从具体业务上下文打开，如凭证、单据、报表单元格、结账检查项或审批记录；入口需要携带来源对象 ID、账套、期间和风险等级。首页只放轻量入口或待办提示，不放大面积常驻对话区。
- Agent 草稿生成入口必须携带目标草稿类型、来源对象、账套、期间、证据引用和风险等级。候选草稿保存前必须经过字段级人工确认；未匹配主数据、LLM Schema 错误和 dry-run 失败时不得保存为系统草稿。

## 10. 前端实现顺序

推荐实现顺序：

1. 搭建 Web App 外壳：布局、导航、账套/期间上下文、权限占位。
2. 建立通用组件：列表页、筛选条、状态标签、详情抽屉、审批确认页。
3. 实现 Phase 1 总账 MVP 页面。
4. 根据 API schema 补齐 Phase 2-4 单据页面。
5. 独立建设 Phase 5 报表设计器。
6. 独立建设 Phase 6 Agent 中心与 Replay。

## 11. UI 验收清单

进入前端开发或验收时，至少检查：

- 每个 Phase 的核心闭环都有页面入口。
- 每个正式写操作都有预览、确认、审计提示。
- 凭证、单据、账簿、报表可以互相联查。
- 权限不足、期间关闭、已锁定状态不会出现可执行按钮。
- 表格支持筛选、排序、分页和导出。
- 移动端只暴露审批、查看和提醒，不暴露复杂录入。
- Agent 相关页面明确区分建议、dry-run、审批和执行。
- Agent 草稿生成页面明确区分模型建议、系统匹配结果、人工修改和最终系统草稿。
- 页面风格保持简洁商务，不出现营销式视觉噪音。
